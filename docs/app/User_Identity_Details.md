# 認証ユーザー機能仕様 (User Identity Details)

本ドキュメントでは、Ultraviolet AI Chatにおけるシステムへのアクセス権およびデータセキュリティの境界となる**「認証ユーザー（Identity）」**について、機能と内部仕様を解説します。

---

## 1. 認証ユーザー (Identity) の定義と役割

認証ユーザー（Identity）は、アプリケーションへのログインを行い、自分のプライベートデータを保護するためのセキュリティ実体です。

*   **スコープ**: システム全体で一意。バックエンドデータベース（SQLite）によって認証が処理されます。
*   **主な役割**:
    1.  ログインおよび新規ユーザー登録（アカウント管理）。
    2.  作成したメモやカスタムエージェントのデータベース上での「所有権（Ownership）」の定義。
    3.  ブラウザ内の `LocalStorage` データに対する名前空間（Namespace）の画定。

---

## 2. 認証のライフサイクル

### A. 新規ユーザー登録 (Sign Up / Register)
*   **インターフェース**: ログイン画面の「新規登録」タブから実行します。
*   **入力項目**: ユーザー名（半角英数字）、パスワード、デフォルトアバター絵文字。
*   **処理**: バックエンドでユーザー名の重複チェックを行い、パスワードは SHA-256 でハッシュ化した上で、SQLite の `users` テーブルに保存します。

### B. ログイン (Login)
*   **インターフェース**: ログイン画面の「ログイン」タブから実行します。
*   **処理**: 入力されたユーザー名とパスワードを照合し、認証に成功すると、暗号署名された **JSON Web Token (JWT)** と共に、ユーザー名、権限ロール、アバター絵文字がフロントエンドに返却されます。
*   **クライアント状態更新**: ログイン成功時に返却された JWT をフロントエンドの LocalStorage (`auth_token`) に保存します。直後に `initializeStateForUser()` が作動し、該当ユーザー固有の LocalStorage 名前空間から設定や履歴が復元されます。

### C. ログアウト (Logout)
*   **処理**: ログイン情報（`auth_username` 等）およびメモリ上のセッション・メッセージ状態をリセットします。
*   *注意*: ユーザー固有の LocalStorage 内のデータは、次回ログイン時の再開のためにブラウザ側に保持されたままとなります。

---

## 3. データベース（SQLite）によるセキュリティ隔離

バックエンドは `app.db` で一元管理されていますが、認証ユーザーごとの安全なデータアクセスを保証するため、アクセス制御が行われています。

### A. 身元の伝達 (`Authorization` Bearer トークン)
フロントエンドからバックエンド（Hono サーバー）への保護された API リクエストには、LocalStorage に保存された JWT が `Authorization: Bearer <token>` HTTP ヘッダーとして自動的に付与されて送信されます。

### B. バックエンドによる認証検証 (Hono JWT Middleware)
バックエンド（Hono サーバー）は、`/api/memos`, `/api/agents`, `/api/chat` などの保護されたルーティングにおいて Hono 標準の `jwt` ミドルウェアを使用して署名と有効期限（7日間）を検証します。
検証された JWT のペイロード（`c.get('jwtPayload')`）から安全に `username` を抽出し、下流のデータベース操作等のオーナー（Owner）識別子として使用します。これにより、クライアントから任意のユーザー名を指定した偽装（ヘッダ改ざんによるスプーフィングバイパス）を完全に防止します。

### B. レコードレベルのデータ隔離
SQLite の `memos` テーブルおよび `agents` テーブルには `owner` カラムが設定されています。
*   **データ取得 (Read)**
    クエリ時に `owner IS NULL`（システム初期データ・共通設定）または `owner = ?`（ログインユーザー名）の条件を強制適用し、他人のデータが絶対に混入しないようにします。
    ```sql
    SELECT * FROM memos WHERE owner IS NULL OR owner = ?
    ```
*   **データ保存 (Write/Update)**
    保存時には、検証済み JWT から抽出した `username` の値を `owner` カラムに書き込みます。
*   **データ削除 (Delete)**
    削除クエリにも `owner` 条件を含め、自分が所有する（または共通の）データのみ削除可能としています。
    ```sql
    DELETE FROM memos WHERE id = ? AND (owner IS NULL OR owner = ?)
    ```

## 4. LocalStorage の名前空間（Namespace）の分離

同じブラウザを使用する異なるアカウント間で API キーや会話ログが混ざるのを防ぐため、LocalStorage キーに対してプレフィックスを自動付与しています。

### A. キープレフィックスの生成
フロントエンドストアの `getAuthKey(key)` ユーティリティにより、すべての個人データ用キーの先頭に `${authUsername}_` が動的に付与されます。
```typescript
export const getAuthKey = (key: string): string => {
  const user = authUsername();
  return user ? `${user}_${key}` : key;
};
```

### B. 分離されるデータ
*   Gemini & Tavily API キー
*   会話セッション一覧 (`chat_sessions`)
*   個々のセッションのメッセージ履歴 (`chat_history_${sessionId}`)
*   ユーザー固有のアクティブなペルソナ情報、選択中のエージェントID等

---

## 5. デフォルトシードアカウント (Default Accounts)

システム初期起動時に、`users` テーブルに以下のテスト用アカウントが自動的にインサート（シード）されます。

1.  **管理者ユーザー**
    *   ユーザー名: `admin`
    *   パスワード: `admin`
    *   ロール: `admin`
    *   アバター: `👑`
2.  **一般ユーザー**
    *   ユーザー名: `user1`
    *   パスワード: `user1`
    *   ロール: `user`
    *   アバター: `👤`
